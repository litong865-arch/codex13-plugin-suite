# 给技术小白的 Codex 13 插件项目流说明书

不是让小白变成程序员，而是让小白能把项目流程说清楚。

版本：2026-06-19 小白版

## 一句话定位

这套插件把 Codex 项目开发拆成 13 个可执行闸门：先判断方向，再写规格，再开发验收，最后上线交付和商业验证。

| 适合人群 | 适合项目 | 核心用法 |
| --- | --- | --- |
| 技术小白、独立开发者、AI 工具使用者 | AI Agent、SaaS、自动化工具、插件开发、网页应用、小程序原型 | 先问项目总控闸门，再按阶段调用对应插件 |

## 1. 为什么做这套插件

这套插件的初衷很简单：我自己本身也是技术小白，但我想用 Codex 做完整项目。真正卡住我的，经常不是某一行代码，而是不知道下一步该问什么、该先查什么、什么时候该验收、什么时候不能上线。

所以这 13 个插件不是为了显得插件很多，而是为了把项目流程变成小白能照着做的顺序。它更像一个项目指挥清单：先总控，再分派；先检查，再开发；先验收，再交付。

> 小白最需要的不是更多术语，而是下一步该做什么、该把什么问题问清楚。

## 2. 小白使用前先记住 3 个原则

- 不要一上来就让 Codex 写代码。先让项目总控闸门判断当前阶段。
- 不要只相信“已完成”。开发后要用 Codex 执行验收器检查按钮、数据、接口和主线流程。
- 不要本地能跑就直接交付。上线前要检查密钥、环境变量、构建、回滚和小白手册。

## 3. 最快上手步骤

1. 先用一句话描述项目：我要给谁解决什么问题，最终交付什么东西。
2. 把这句话发给 Codex，并要求先调用“项目总控闸门”。
3. 根据总控输出，按顺序调用 1 到 3 个插件，不要同时乱调一堆。
4. 等规格、接口、成本、安全这些前置问题清楚后，再让 Codex 开始写代码。
5. Codex 说完成后，用“代码漂移巡检器”和“Codex 执行验收器”检查。
6. 准备交付前，用“上线发布守护器”和“小白执行手册生成器”收尾。
7. 如果要商业化，再用“用户访谈生成器”和“竞品分析器”验证真实需求。

可直接复制给 Codex：

```text
我是技术小白，想做一个【在这里写你的项目】。
请不要直接写代码。先使用项目总控闸门判断：
1. 我现在处于项目哪个阶段？
2. 现在最应该调用哪几个插件？
3. 哪些问题如果不解决，会导致项目烂尾？
4. 请给我下一步可复制的提示词。
```

## 6. 项目总控闸门怎么调度

项目总控闸门不是替你写所有代码的万能按钮。它的价值是先判断项目卡在哪个阶段，再把任务交给合适的插件或工具。

| 可调度能力 | 适合场景 | 小白怎么理解 |
| --- | --- | --- |
| 插件调度 | 压力测试、规格、接口、成本、安全、验收、上线、商业验证 | 让 Codex 按阶段找专业插件，不靠你自己记顺序。 |
| 内置浏览器 | 本地网页、localhost、截图、点击按钮、检查页面 | 适合检查本机项目页面是不是真的能用。 |
| 谷歌浏览器 | 需要登录状态、已有账号、远程网站、平台后台 | 适合检查必须依赖你 Chrome 登录状态的页面。 |
| 电脑操作 | Windows 弹窗、安装器、桌面软件、文件选择器 | 适合处理普通浏览器做不了的桌面操作。 |
| 子线程 | 竞品、接口、规格、验收、安全等任务可以并行推进 | 适合大项目，让多个方向同时查，但最后由总控汇总。 |

> 总控能调用什么，取决于当前 Codex 环境里是否已经安装并启用了对应插件或工具。它负责判断和分派，最终执行仍由 Codex 当前可用能力完成。

## 8. 安装包小白安装说明

安装包文件名是：`Codex项目开发插件13件套安装包.zip`。发给其他用户时，可以让对方按下面步骤操作。

1. 把 zip 压缩包保存到电脑任意位置，比如桌面。
2. 右键压缩包，选择“全部解压”或“解压到当前文件夹”。
3. 打开解压出来的 `codex-project-plugin-suite` 文件夹。
4. 双击 `Install-Double-Click.bat`。
5. 如果 Windows 弹出安全提示，确认这是你信任的安装包后继续运行。
6. 看到安装完成提示后，关闭窗口并重启 Codex。
7. 重启后进入已安装插件列表，查看 13 个插件是否出现。
8. 如果只想先用一个，优先选择“项目总控闸门”。

验证是否安装成功，可在 Codex 里问：

```text

```

## 9. 小白指挥 Codex 的常用话术

| 场景 | 直接复制的话 |
| --- | --- |
| 刚有想法 | 我想做【项目】，我是技术小白。请先用项目总控闸门判断下一步，不要直接写代码。 |
| 怕项目没人要 | 请用决策压力测试器和用户访谈生成器，帮我判断这个项目有没有真实需求。 |
| 准备开工 | 请先用产品可行性闸门和规格生成器，把 MVP 主线、页面、数据、接口和验收标准写清楚。 |
| 涉及第三方平台 | 请用接口/平台审查器检查这个平台是否允许做，有没有官方 API 和风控限制。 |
| 涉及 AI 调用 | 请用 AI 成本估算器估算单次、每天、每月成本和延迟。 |
| Codex 改了很多轮 | 请用代码漂移巡检器检查项目是否越写越乱，并给我修复顺序。 |
| Codex 说完成 | 请用 Codex 执行验收器跑主线流程，证明功能真的能用。 |
| 准备发给别人 | 请用上线发布守护器和小白执行手册生成器，检查交付风险并写出照做步骤。 |

## 10. 常见错误和处理办法

| 常见错误 | 为什么危险 | 正确做法 |
| --- | --- | --- |
| 一上来就写代码 | 需求、接口、成本、安全都没确认，后面容易返工。 | 先问项目总控闸门，让它判断阶段和插件顺序。 |
| 只看 Codex 说完成 | 可能按钮是假的、数据没保存、接口没接通。 | 用 Codex 执行验收器跑真实主线。 |
| 本地能跑就交付 | 别人电脑可能缺环境变量、依赖、说明和回滚方案。 | 用上线发布守护器和小白执行手册生成器收尾。 |
| 相信平台一定能自动化 | 平台可能没有 API，或者不允许自动发布、抓取、评论。 | 涉及平台先走接口/平台审查器。 |
| AI 功能不估成本 | 上线后可能额度爆掉、延迟过高、按量费用失控。 | 先用 AI 成本估算器做预算。 |
| 项目越改越乱还继续加功能 | 目录、状态、模拟数据会混在一起，越到后面越难修。 | 用代码漂移巡检器先清理再继续。 |

## 11. 交付前检查清单

- 已经用项目总控闸门确认当前阶段和下一步。
- 已经有清晰规格，不是靠 Codex 猜。
- 涉及平台能力时，已经确认官方接口和规则限制。
- 涉及 AI 调用时，已经估算成本、延迟、额度和降级方案。
- 开发后已经做代码漂移巡检。
- Codex 说完成后，已经做执行验收。
- 上线前已经检查环境变量、密钥、构建、测试、README 和回滚。
- 交付给别人前，已经生成小白执行手册。
- 商业化前，已经做用户访谈和竞品分析。

> 真正提升效率的，不是让 AI 更自由，而是让流程更可控。

## 安全说明

这个仓库包含“安全闸门”插件，因此源码和模板里会出现 `token`、`secret`、`api key` 等安全审查关键词。这些词用于描述安全检查规则、风险提示和占位符，不是实际密钥。

发布前已做基础敏感信息检查，并把容易被误判成真实密钥的假示例改成了普通占位符。公开仓库后，仍建议开启 GitHub secret scanning，作为后续版本的自动防线。
