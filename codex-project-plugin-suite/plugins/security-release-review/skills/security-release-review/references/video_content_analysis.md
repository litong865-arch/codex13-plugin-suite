# 上传视频内容分析：能跑只是第一步，上线前要拆风险

> 这是根据用户上传视频画面与字幕整理出的结构化分析，供 release-safety-gate skill 作为设计依据。

## 视频主旨

视频围绕一个“明星 App / Vibe Coding 项目”的案例展开。它强调：AI 现在很擅长把功能快速做出来，非程序员也可以完成一个看似能用的 App；但 demo 能跑不等于可以公开上线。公开上线后，真实用户、脚本、广告党和滥用者会把功能当成通道使用，原本“方便用户”的设计可能变成攻击入口、成本黑洞或合规事故。

核心观点：

- 先承认“能做出来已经不容易”。
- 但“跑通功能也跑出风险”。
- 上线前不能只问“功能实现了吗”，还要问“坏人会怎么用它”。
- 让 AI 不只是写功能，还要模拟安全审计员。
- 公共空间需要“门锁和消防通道”：限流、审核、下架、报警、回滚和追溯。

## 视频拆出的五类高危点

### 1. 方便登录可能变短信发射器

视频指出，验证码短信能发出来不代表安全。对用户来说少一步很方便，对攻击者来说多了一条通道。每一条短信背后都是成本，如果没有限速、预算和报警，接口可能被滥用成“免费短信发射器”。

应对策略：

- 手机号、IP、设备、账号、会话多维度限速。
- 每日/每小时发送上限。
- 供应商预算、账单告警、异常峰值报警。
- 验证码冷却时间、尝试次数限制、短有效期。
- 防用户枚举和机器人批量注册。

### 2. UGC 一公开就有占位风险

头像、昵称、简介、评论、留言、许愿墙等看似很小的功能，都可能被用于广告引流、违规内容占位、刷屏和敏感信息传播。视频强调：不能默认用户都是善良的，产品只要有一点热度，就会有人尝试塞引流和违规内容。

应对策略：

- 公开内容先进入审核或风控队列。
- 高风险内容先隐藏，审核通过后再展示。
- 对外链、敏感词、重复内容、垃圾广告、恶意 HTML/Markdown 做过滤。
- 提供后台审核、下架、封禁、审计日志和申诉机制。
- 对公开资料和评论做频率、长度、链接数量限制。

### 3. 图片上传会变公共图床

视频指出，图片上传后通常会产生一个 URL。如果这个 URL 是永久公开的，平台可能被人拿来当图床；如果图片违法违规，域名和存储桶也可能背锅。图片风险不是“能上传”这个点，而是完整链路：大小、格式、内容审核、权限、防盗链、隔离和删除。

应对策略：

- 服务端限制文件大小、类型、MIME、magic bytes 和图片尺寸。
- 去除 EXIF，防压缩炸弹、路径穿越和任意文件上传。
- 上传后先进入隔离区，审核通过后再公开。
- 存储桶默认私有，使用签名 URL、过期时间、权限校验。
- 防盗链、访问配额、生命周期、违规投诉处理。

### 4. AI 聊天不是保险箱

视频提醒：角色设定再可爱，AI 也可能被用户通过改写问题、套话、提示注入等方式诱导说出隐藏规则、系统提示或内部指令。不能把敏感规则、密钥或关键权限交给模型“自觉保密”。

应对策略：

- 系统提示和密钥只保存在服务端，不能放到前端。
- 输入过滤、输出审核、敏感内容拦截。
- 隐藏规则、内部配置、密钥、工具权限不交给模型自行决定是否泄露。
- 对 prompt injection、system prompt extraction、jailbreak 做测试。
- 工具调用最小权限，有审计、限速、成本控制和 kill switch。

### 5. Demo 爽感不等于公开世界可靠

视频把 demo 和公开上线做了对比：自己试用很顺，但真实用户、脚本、广告党、无聊但有技术的人都会进入。上线前必须把视角从“产品能不能跑”切换为“坏人怎么用”。

应对策略：

- 上线前按短信、UGC、图片、AI、权限逐项压测和审查。
- 后台必须能下架、封禁、报警、回滚和追溯。
- 对成本型功能设置预算和异常检测。
- 让 AI 模拟审计员，而不是只让 AI 写功能。

## 对 skill 设计的直接启发

这个 skill 的默认工作方式是：

1. 先画出公开入口和信任边界。
2. 自动扫描仓库中的高危信号。
3. 逐项套用短信、UGC、图片、AI、权限、密钥、监控门禁。
4. 用“坏人视角”提问并找证据。
5. 输出可执行的 `SECURITY_RELEASE_REVIEW.md`。
6. 发现阻断级问题时明确给出 `BLOCK`，而不是为了上线而淡化风险。

## 一句话总结

能跑只是第一步；上线前先拆一遍，看看坏人会怎么用它。
