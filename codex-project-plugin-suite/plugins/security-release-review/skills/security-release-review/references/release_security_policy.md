# Release Security Policy：上线前安全策略

本策略用于 AI/Vibe Coding 项目、Web App、移动后端、SaaS、小程序、社区、AI 聊天应用、UGC 应用在公开上线前做发布门禁检查。

## 0. 发布结论标准

### PASS

可以上线。没有未缓解的 `BLOCKER` 或 `HIGH`；中低风险有 owner、时间表和监控。

### PASS_WITH_CONDITIONS

可以在满足明确条件后上线。条件通常包括：上线前修复高风险、开启限流/审核/告警、只灰度给白名单用户、关闭高风险功能。

### BLOCK

不建议公开上线。存在至少一个未缓解 `BLOCKER`，或多个 `HIGH` 叠加导致真实滥用/数据泄露/成本失控/合规事故概率高。

## 1. 阻断级风险定义

出现以下任一情况，默认 `BLOCKER`：

- 真实密钥、私钥、数据库密码、供应商 token 入库或暴露给前端。
- 匿名或普通用户可无限制触发短信、邮件、支付、AI token 消耗等成本型资源。
- 上传文件可直接形成永久公开 URL，且无大小/类型/内容审核/权限控制。
- 公开 UGC 提交即展示，且可包含链接、头像、简介、评论等高滥用字段，无审核和下架手段。
- AI 模型可访问敏感工具、内部数据或系统提示，且无服务端权限控制、审计和输出审核。
- Mutating API 无认证/授权，对象级权限缺失，存在明显越权修改/删除/读取风险。
- 管理员功能暴露给非管理员，或只依赖前端隐藏。
- 生产环境 debug、测试后门、默认管理员账号、公开错误堆栈泄露敏感信息。
- 无法下架、封禁、回滚、报警或追溯关键公开内容。

## 2. 身份、验证码与成本型接口

### 必须具备

- IP、账号、目标手机号/邮箱、设备/指纹、会话维度限速。
- 冷却时间、尝试次数、验证码有效期和重放保护。
- 全局预算、供应商账单阈值、异常峰值报警。
- 防枚举提示，避免暴露“手机号是否注册”。
- 关键接口只在服务端调用供应商，前端不能持有供应商密钥。

### 建议具备

- 风险评分、验证码/挑战、人机验证、黑名单/灰名单。
- 注册、重置密码、绑定换绑等高风险操作有更严格策略。
- 供应商失败时有降级方案和异常日志。

### 常见坏味道

- `sendSms(phone)` 公开 API 无 auth/rate limit。
- `Math.random()` 生成 OTP。
- 发送成功/失败文案暴露账号存在性。
- 前端直接调用短信供应商 API。

## 3. UGC、资料页与公开内容

### 必须具备

- 公开展示前的审核/风控状态机：`pending`、`visible`、`hidden`、`rejected`。
- 敏感词、垃圾广告、外链、恶意 HTML/Markdown、脚本注入过滤。
- 长度、频率、链接数量、重复内容、批量提交限制。
- 后台审核、下架、封禁、审计日志。
- XSS 防护：服务端清洗或严格转义；富文本使用白名单。

### 建议具备

- 低信任用户默认延迟展示或人工审核。
- 新账号冷启动限额。
- 举报、申诉、内容版本记录。

### 常见坏味道

- 评论/昵称/简介提交即公开。
- 可上传头像但无图片审核。
- 支持 Markdown/HTML 但未清洗。
- 删除只是前端隐藏，后端仍公开可访问。

## 4. 文件与图片上传

### 必须具备

- 大小限制、扩展名白名单、MIME 校验、magic bytes 校验。
- 图片尺寸限制、重编码、EXIF 去除、压缩炸弹防护。
- 私有存储桶，签名 URL，过期时间，访问权限校验。
- 内容审核或隔离区，不合规内容不可公开。
- 防路径穿越、文件覆盖、公开列目录、任意文件执行。
- 生命周期、删除、审计、投诉处理。

### 建议具备

- 防盗链、CDN token、访问配额。
- 病毒扫描、异步审核队列、人工复核。
- 不直接使用原始文件名；使用不可预测对象 key。

### 常见坏味道

- `public-read`、`makePublic()`、永久公开 URL。
- 只检查扩展名不检查 magic bytes。
- 无文件大小限制。
- 上传后立即公开展示。
- 对象 key 使用用户输入文件名。

## 5. AI 聊天、Agent 与模型工具

### 必须具备

- 模型密钥、系统提示、内部策略只保存在服务端。
- 输入过滤、输出审核、敏感信息拦截。
- prompt injection、system prompt extraction、jailbreak 测试。
- 工具调用有 allowlist、参数校验、最小权限、审计日志。
- 成本限速、token 预算、用户级配额、异常报警。
- RAG 有租户隔离、权限过滤和引用溯源。

### 建议具备

- 针对高风险话题的人工升级或拒答策略。
- 模型输出落库前二次审核。
- 针对系统提示泄露的回归测试集。
- kill switch：一键禁用 AI 聊天或敏感工具。

### 常见坏味道

- 前端包含 `OPENAI_API_KEY` 或模型供应商 key。
- 把“不要泄露规则”只写在 prompt 里。
- AI 输出直接公开展示。
- Agent 可执行 shell、发短信、改数据库但没有审批。

## 6. API、权限与会话

### 必须具备

- 所有写接口都有认证、授权、对象级权限。
- 管理员接口服务端强校验角色，操作有审计日志。
- 防 CSRF、CORS 配置收敛、Cookie 设置 Secure/HttpOnly/SameSite。
- 防 IDOR：不能通过修改 ID 访问别人的资源。
- 数据库查询参数化，避免拼接 SQL。
- 返回给前端的数据最小化，敏感字段不出后端。

### 建议具备

- 关键操作二次确认。
- 会话吊销、设备管理、异常登录提醒。
- 统一错误处理，避免堆栈泄露。

### 常见坏味道

- 前端隐藏按钮等于权限控制。
- API 根据 `userId` 参数信任客户端。
- `Access-Control-Allow-Origin: *` 配合 credentials。
- 生产环境返回详细 stack trace。

## 7. 密钥、配置与供应链

### 必须具备

- 密钥不入库、不入前端、不进日志。
- `.env.example` 使用占位符，真实 `.env` 在 `.gitignore`。
- CI/CD secret 最小权限，部署环境隔离。
- 生产关闭 debug/test mode/mock admin。
- 依赖锁文件存在，基础依赖做漏洞扫描。

### 建议具备

- 定期轮换密钥。
- 分环境密钥和访问策略。
- 对依赖新增做 review。

## 8. 监控、后台与事故响应

### 必须具备

- 错误率、请求量、登录失败、验证码发送、上传量、AI token 消耗、审核命中率监控。
- 异常峰值告警和成本告警。
- 后台可下架内容、封禁用户、撤销文件、关闭功能、回滚版本。
- 审计日志记录关键动作：谁、何时、做了什么、影响对象。

### 建议具备

- 灰度发布、白名单、feature flag。
- 事故响应联系人和 runbook。
- 数据保留与删除策略。

## 9. 最小上线安全包

对于小团队或个人项目，至少满足：

- 密钥不在前端和仓库。
- 登录/验证码/上传/AI 调用有限速和预算。
- UGC 和图片先审核或先隐藏。
- 上传文件私有存储，签名访问。
- 所有写接口有服务端认证授权。
- 有后台下架/封禁/关闭功能的能力。
- 有基础日志和异常报警。

这不是“企业级安全”的全部，但可以避免 demo 直接裸奔到公网。
