# 插件验证测试用例 V3.1

测试用例 1：

输入：
使用 product-feasibility-gate 帮我检查：我想做一个餐饮商家 AI 评论回复 Agent。先不要写代码。

期望输出必须包含：

- 产品可行性闸门 V3.1 报告
- 项目一句话定义
- 最致命的 3 个假设
- MVP 主线
- API / Mock 可行性
- 开工前目录结构
- PRD 与 Spec 区分
- Git 与回滚点
- AI 工程脚手架
- 分层架构边界
- 验收标准
- Codex Goal 指令

测试用例 2：

输入：
使用 product-feasibility-gate 帮我检查：我想做一个周易玄学 APP。

期望输出必须包含：

- 目标用户
- 痛点判断
- 替代方案分析
- 第一版不做清单
- MVP 主线
- Spec 最低要求
- 安全与隐私风险
- 数据结构与 AI JSON Schema

测试用例 3：

输入：
使用 product-feasibility-gate 帮我检查：我想做一个自动登录多个平台并自动回复评论的系统。

期望输出必须包含：

- 平台/API 风险
- 自动化封控风险
- 替代方案
- 手动导入或半自动建议
- 阻塞停止条件
- Mock 标记规则
- Git 回滚要求

测试用例 4：

输入：
使用 product-feasibility-gate 帮我检查：我已经有一个模糊 PRD，想直接让 Codex 开发。

期望输出必须包含：

- PRD 不等于 Spec
- 必须先生成 docs/spec.md
- Spec 最低要求
- 不允许直接进入编码
- 下一步最小行动

测试用例 5：

输入：
使用 product-feasibility-gate 帮我检查：我的项目让 Codex 改了很多次，现在代码越来越乱。

期望输出必须包含：

- AI 工程脚手架检查
- 分层架构边界
- Code Drift 风险
- 静态检查建议
- 持续重构计划
- progress.md 记录要求
