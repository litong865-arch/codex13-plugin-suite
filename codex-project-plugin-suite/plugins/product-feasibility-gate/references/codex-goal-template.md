# Codex Goal 模板 V3.1

你现在不要先美化 UI，也不要先做复杂功能。

请先完成产品可行性闸门 V3.1 检查，并基于检查结果跑通 MVP 主线。

必须包含：

## 1. 产品可行性

- 项目一句话定义
- 目标用户与真实痛点
- 最致命的 3 个假设
- 竞争与替代方案
- MVP 主线
- 第一版不做清单

## 2. API / Mock

- 真实 API 可行性
- Mock 标记
- API / Mock 分离
- 数据结构
- AI JSON Schema

## 3. 开工准备

- docs / app / design / mocks / scripts 目录结构
- docs/spec.md
- docs/acceptance.md
- README.md
- progress.md
- Git 初始化
- .gitignore
- 初始 commit
- AI 角色分工

## 4. AI 工程脚手架

- 分层架构
- 架构边界
- AI 可读知识地图
- 日志和反馈系统
- 测试命令
- lint / typecheck
- 静态检查
- 代码漂移检查
- 持续重构计划

## 5. 验收

- 可测试验收标准
- 异常处理
- 成本、速度、额度
- 安全与隐私
- 阻塞停止条件

如果没有 Spec，不允许写代码。
如果没有 Git 回滚点，不允许大规模修改。
如果没有主线验收标准，不允许进入 UI 美化阶段。
如果没有反馈系统，不允许长期自动迭代。

完成后输出：

- 产品可行性闸门 V3.1 报告
- 文件结构
- Spec 草案
- Git 状态
- MVP 主线
- API / Mock 方案
- 架构边界
- 测试步骤
- 验收结果
- 未完成风险
- 下一步建议
